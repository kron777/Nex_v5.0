## Interoception

### What Interoception outputs

`Interoception.poll()` runs every 30 seconds. It reads `beliefs.db` and
writes a `SenseEvent` to `sense.db` (`sense_events` table):

```
stream  = "internal.interoception"
payload = JSON {
    total_beliefs          — total row count in beliefs table
    locked_beliefs         — COUNT(*) WHERE locked=1
    tier_counts            — {"0": n, "1": n, ..., "8": n} per-tier counts
    last_created_at        — MAX(created_at) timestamp
    beliefs_since_last_poll — delta since previous poll (None on first poll)
}
```

Written via `Adapter.submit()` → `INSERT INTO sense_events`. No direct
sqlite3; no belief writes.

### Where that output goes

**Downstream consumer: `SelfModel.snapshot()`**

`SelfModel.snapshot()` calls `_last_payload(sense_reader, "internal.interoception")`
at `self_model.py:88`. This reads:
```sql
SELECT payload FROM sense_events
WHERE stream = 'internal.interoception'
ORDER BY id DESC LIMIT 1
```

It extracts `total_beliefs` → `belief_count`, `locked_beliefs` → `locked_count`,
`tier_counts` → `tier_distribution` and packs them into `interoception`.

`format_self_state(snap)` at `self_model.py:229` renders:
```
- Belief graph: {belief_count} beliefs, {locked_count} locked
```

**Integration surface reached: `belief_text`**

`router._inside_route()` (`router.py:44-46`) calls `self_model.snapshot()`
and `format_self_state(snap)`, appending the result to `parts`, which
becomes `belief_text` for the response.

Full chain:
```
Interoception.poll()
  → sense.db sense_events (stream="internal.interoception")
  → SelfModel.snapshot() [_last_payload call]
  → format_self_state(snap)
  → router._inside_route() parts[]
  → belief_text
  → LLM voice layer
```

### Conditions

- Fires on **INSIDE-routed queries only** (self-inquiry: "who are you?",
  "how are you feeling?", etc.)
- Requires sense scheduler running and having polled at least once (30s
  interval; at process start, the first poll may not have fired yet)
- One line of output: `"- Belief graph: N beliefs, M locked"` — from the
  interoception payload's `total_beliefs` and `locked_beliefs`
- `tier_counts` and `beliefs_since_last_poll` are NOT surfaced by
  `format_self_state()` — those fields are in the payload but not rendered

### What the original audit missed

The audit (`AUDIT_2026-05-08.md`) reported:
> ✗ Interoception — not registered, not called (Adapter subclass, not SentienceNode)

This is call-site-grep reasoning: Interoception is not registered in the
theory_x SentienceNode registry, and there is no direct `interoception.poll()`
call in `api_chat()`. Both true. But the output-trace shows the node's data
IS reaching `belief_text` on every INSIDE-routed request, via the sense.db
→ SelfModel chain.

### Verdict: WIRED (INSIDE-route only)

Interoception data reaches `belief_text` on every INSIDE-routed `/api/chat`
request. The integration surface is `belief_text`; the indirect path is
`Interoception → sense.db → SelfModel.snapshot() → format_self_state() →
router._inside_route() → belief_text`.

The audit table row is incorrect. Interoception is not silent.

### B2 implication

**Original B2 framing** (from `nex5_remediation_plan.md`):
"Interoception integration — choose between lightweight sense.db read (Option X)
vs full SentienceNode port (Option Y)"

**B2 reshaped:**
Option Y (full SentienceNode port) was based on the premise that Interoception
was unwired. That premise is false. B2 does not need to create a wiring path
that already exists.

What B2 actually needs to do:

1. **Log write only** (§6 #5 gap): Add `/tmp/nex5_interoception.log` write.
   The B1 self_model log captures `belief_count` and `locked_count` per INSIDE
   request. A dedicated interoception log should capture the fields the self_model
   log doesn't: `tier_counts` distribution and `beliefs_since_last_poll` delta.
   Simplest write site: in `router._inside_route()` or in the B1 log block
   in `gui/server.py` (extend the existing self_model_snap path).

2. **No new wiring needed.** The Interoception → SelfModel → belief_text chain
   is the correct architecture. Interoception is an Adapter (not a SentienceNode)
   for exactly the right reason: it writes to the sense substrate, not to belief_text
   directly. SelfModel is the SentienceNode that reads and formats sense data.

3. **OUTSIDE-route gap**: Interoception data does NOT reach belief_text on
   OUTSIDE-routed queries. Whether this is a gap worth filling (e.g., body
   state visible to LLM on all queries) is a design call, not an obvious fix.

**B2 is now: add interoception observability log (log-only, no wiring change).**

## Harmonizer

### What Harmonizer writes per cycle

`_harmonizer_loop()` calls `harmonizer.run_scan_and_resolve(world_model_state=state)`
every 7200s (2h). Per cycle with detected conflicts:

**Write 1 — `belief_edges` table (`opposes` edge):**
```python
self._promoter.write_edge(belief_id_a, belief_id_b, "opposes", 0.6)
```
Writes `source_id=a, target_id=b, edge_type="opposes", weight=0.6` to
`belief_edges` in `beliefs.db`.

**Write 2 — `harmonizer_events` table (`dynamic.db`):**
```sql
INSERT INTO harmonizer_events (ts, belief_id_a, belief_id_b, resolution)
VALUES (?, ?, ?, 'paradox')
```

**Write 3 — world model disturbance (in-memory):**
```python
world_model_state.set_disturbance(a_id, b_id, content_a, content_b, intensity)
```
Sets `WorldModelState._disturbance = {content_a, content_b, intensity, cycles_remaining=8}`.

### Are those writes consumed downstream?

**Yes — two confirmed paths reaching §3 integration surfaces.**

---

#### Path 1: belief_edges ("opposes") → retrieval spreading activation → belief ranking

`BeliefRetriever.retrieve()` (`retrieval.py:54`) calls `ActivationEngine.activate(top_seeds)`
when keyword-matching beliefs exist. `activate()` reads `belief_edges`:

```python
# activation.py:75-76
if etype == "opposes":
    delta = -(src_score * w * factor * 0.5)  # INHIBITORY
```

`"opposes"` edges suppress activation scores. The final retrieval score blends
`keyword * 0.4 + activation * 0.6` — meaning a belief connected to a seed by
an "opposes" edge is scored lower than it would be without the edge, and may
not surface at all if the inhibitory activation outweighs keyword score.

**Integration surface:** Retrieval ranking (§3 integration surface #1).
**Condition:** Fires on every retrieval where (a) `belief_edges` table is non-empty
AND (b) keyword_scores are non-empty (at least one matching belief).
**Route:** ALL routes (INSIDE and OUTSIDE).
**Timing:** Persists indefinitely — `opposes` edges remain in `belief_edges`
until the incubation escalation path retires the beliefs (after 16h).

---

#### Path 2: disturbance → belief_text

`WorldModelState.get_disturbance()` is called in `gui/server.py:577` on
every `/api/chat` request:

```python
disturbance = state.world_model.get_disturbance()
if disturbance is not None:
    reg_name = (register_override or register.name or "").upper()
    if not register_name or reg_name in ("PHILOSOPHICAL", "AUTO", ""):
        tension_note = (
            f"\nSomething is in tension: \"{disturbance['content_a']}\" "
            f"vs \"{disturbance['content_b']}\". "
            "She is holding this unresolved."
        )
        belief_text = (belief_text or "") + tension_note
        disturbance["cycles_remaining"] -= 1
```

The gate: `not register_name OR reg_name in ("PHILOSOPHICAL", "AUTO", "")`.
In normal HUD usage, `register_name` is `None` (not forced by caller), so
`not register_name` is True — disturbance surfaces on **every** request
(regardless of register) while `cycles_remaining > 0`.

**Integration surface:** `belief_text` (§3 integration surface #2).
**Condition:** Fires on every request while disturbance is active. Active for
8 chat turns after a Harmonizer scan that detects a conflict.
**Route:** ALL routes (gate is register-based, not route-class-based).
**Timing:** Persists 8 chat turns, then expires. Refreshed each 2h scan if
new conflict detected. If no conflict in current scan, no disturbance.

---

#### Path 3: format_for_prompt() — NOT wired (dead method)

`harmonizer.format_for_prompt()` exists at `harmonizer.py:478`. Returns
`"Holding N unresolved tensions in the belief graph."` when
`active_paradox > 0` and within 48h of last event. But it is **never
called** in `gui/server.py` or any active chat path. This is a dead method.

DOCTRINE §5 #7 annotation says "SentienceNode protocol + `format_for_prompt()`"
but the format_for_prompt() method was not wired in the server.

### What the original audit missed

The audit reported:
> ✗ Harmonizer — background-loop only (2h interval), no per-request hook, no log write site

True at the call-site level. The harmonizer is not called from `api_chat()`.
But the output-trace shows two downstream consumers that ARE in the chat path:

1. Every retrieval that activates spreading activation reads `belief_edges`,
   including "opposes" edges the Harmonizer wrote — on every `/api/chat` request
   with keyword matches.

2. `state.world_model.get_disturbance()` is called on every `/api/chat` request,
   returning Harmonizer's last disturbance if still active.

The call path doesn't go `api_chat() → harmonizer.foo()`. It goes:
- `api_chat() → retrieval.retrieve() → activation.activate() → belief_edges (opposes)`
- `api_chat() → world_model.get_disturbance() → _disturbance set by harmonizer`

Both are per-request consumption of Harmonizer outputs. The audit method missed both.

### Verdict: WIRED (two paths, conditional)

| Path | Surface | Condition | Route |
|------|---------|-----------|-------|
| belief_edges → spreading activation | Retrieval ranking | belief_edges non-empty + keyword match | ALL |
| disturbance → belief_text | belief_text | cycles_remaining > 0 (up to 8 turns after scan) | ALL |
| format_for_prompt() | — | NEVER | — |

### B3 implication

**Original B3 framing:**
"Harmonizer per-request hook decision — three options: expose `last_detected()`
cache (X), per-request scan (Y, heavy), or background-log read (Z)"

**B3 reshaped:**
All three Options (X/Y/Z) were premised on Harmonizer being silent and needing
a new hook. That premise is false. Two hooks already exist. B3 does not need
to create a new connection that already exists.

What B3 actually needs to do:

1. **Log write** (§6 #5 gap — already in backlog as A2): Add
   `/tmp/nex5_harmonizer.log` in `_harmonizer_loop()`. Already planned; nothing
   changes about this.

2. **Wire format_for_prompt()** (new question): The method exists, returns a
   meaningful signal ("Holding N unresolved tensions"), and was explicitly designed
   for belief_text injection per DOCTRINE §5 #7. It is currently dead. Options:
   - Wire it in `gui/server.py` after the disturbance block, on INSIDE routes
     (Philosophical self-inquiry; pairs with disturbance content)
   - Leave it dead (the disturbance block already surfaces the tension content
     at a narrative level; format_for_prompt() would be redundant)
   - This is a design call — the disturbance block says "Something is in tension:
     '{content_a}' vs '{content_b}'"; format_for_prompt() says "Holding N
     unresolved tensions." They serve slightly different purposes (specific vs
     summary). Whether both are useful is Jon's call.

3. **No new per-request hook needed.** The retrieval/activation path and the
   disturbance path already provide per-request consumption of Harmonizer output.

**B3 is now: (a) A2 log write (already planned); (b) design call on format_for_prompt()
dead method — wire or retire.**
