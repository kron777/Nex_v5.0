# Tag Protocol — Specification

Foundational spec per DOCTRINE §1. Defines the comprehensive
tag system across NEX's cognition substrate. Written before
implementation. Build begins in a separate session.

Created: 2026-05-11. Design decisions locked in Phase 25b-redesign
spec session with Jon. Tags are a compressed semantic language —
one vocabulary across all surfaces of the substrate.

---

## §1 Purpose

The tag protocol gives every cognition entity (belief, open
problem, review-queue item, and future surfaces) a shared
semantic categorization layer. Tags are the compressed
vocabulary NEX uses to know what an entity is *about* when
it is re-referenced.

The principle: **same language across the board.** A tag
means the same thing on a belief that it means on a problem,
on a review-queue row, and on any future taggable surface.
No per-table dialects. No dimensional sub-vocabularies. One
language.

The function: when NEX encounters an entity (during
retrieval, gate evaluation, prompt-building, narrative
construction, drive detection — anywhere cognition uses the
substrate), the entity's tags arrive with it as a unified
context payload. NEX can form a position on the entity
without further substrate queries, because the tags carry
the essential semantic context inline.

Tags are auto-generated by NEX from entity content. The
vocabulary evolves through use, not through curation gates.

**What tags are not:**

Tags are not free-text descriptions. They are normalized,
short, vocabulary-discipline-bearing strings.

Tags are not status flags. State (open / closed / flagged /
etc.) is tracked in dedicated columns or by table membership,
not in tags.

Tags are not the only context NEX uses. They sit alongside
existing structure (branch_id, confidence, problem_id,
embeddings) — they don't replace any of it.

## §2 Storage

Inline JSON column on each taggable table.

**Schema pattern:**

```sql
ALTER TABLE <table> ADD COLUMN tags TEXT NOT NULL DEFAULT '[]';
```

**Format:** JSON array of normalized lowercase strings.
Empty default: `'[]'`. Never NULL.

**Initial surfaces:**

- `beliefs.tags` (in `beliefs.db`)
- `open_problems.tags` (in `conversations.db`)
- `review_queue.tags` (in `conversations.db`, added by Phase 25b)

**Future surfaces** (to be added as cognition surfaces adopt
the protocol):

- arc notes, held thoughts, drives, narrative entries,
  novel_association_log, gate_decisions metadata,
  fountain output, etc.

**Migration:** applied via the existing `_MIGRATIONS`
idempotent pattern in `substrate/init_db.py`. The
`_column_exists` guard handles re-running.

**Why inline JSON:**

- Reads stay self-contained. When NEX reads an entity, the
  tags travel along in the same row. No JOIN, no second
  query, no parsing across tables. Tags arrive with the
  entity as a unified context payload (§1).
- Extends NEX's existing pattern. `branch_id` is a single-value
  categorical label living on the row; tags are the natural
  multi-value generalization.
- No cross-DB friction. NEX's substrate splits across multiple
  SQLite files; a polymorphic tags table would require ATTACH
  gymnastics or per-DB duplication. Inline columns sidestep
  this entirely.
- Migration is tiny. Each relevant table gets one column;
  no new tables, no junction tables, no FK constraints to
  maintain across DBs.

**Cost:** tag-based inverse lookups (`find every belief tagged
'cognition'`) are O(n) scans via SQLite JSON1 functions
(`json_each`, `json_extract`, `LIKE` with the array
serialization). At current scale (~6k beliefs, hundreds of
problems), this is sub-second. Promotion path: a secondary
per-DB `tag_index(entity_type, entity_id, tag)` table can
be added if scale demands faster inverse lookup — inline JSON
stays the source of truth (§10).

## §3 Vocabulary

**Free-form at write time.** NEX (and Jon, when manually
intervening) can coin any new tag at any moment. No
whitelist, no gate, no blocking error on novel strings.

The vocabulary is observable at any time via:

```sql
SELECT DISTINCT value
FROM beliefs, json_each(beliefs.tags)
UNION
SELECT DISTINCT value
FROM open_problems, json_each(open_problems.tags)
-- ... one UNION per taggable table
```

A helper view or function in the tag protocol module
provides this aggregated vocabulary list across all taggable
tables.

**Evolution model:**

The vocabulary grows through use. No curation occurs at
write time. Curation is periodic and out-of-band (§7).

Initial deployment: vocabulary is whatever NEX coins.
Drift, typos, and near-duplicates are accepted as part of
the working state. They will be detected and merged
periodically.

## §4 Generation

**NEX auto-generates tags.** No manual tagging required at
the entity creation path. (Jon may still manually add or
remove tags on individual entities via dedicated operations;
see §5.)

**Mechanism: keyword extraction from content.**

Per §0 doctrine — *substrate solves the reply, LLM speaks* —
tag generation is a substrate operation. No LLM call.

**Pipeline:**

1. **Input:** the entity's primary content field(s):
   - `beliefs`: `content`
   - `open_problems`: `title` + (if present) `description`
   - `review_queue`: inherits from `open_problems` at move time
   - Future surfaces: their content-bearing field(s)

2. **Tokenize:** split on whitespace; lowercase each token.

3. **Strip:** apply the existing `TriggerDetector` stopword
   strip (same one used in TN-1 throw-net retrieval; reuse,
   don't fork).

4. **Filter:**
   - Drop punctuation-only tokens
   - Drop pure-numeric tokens (configurable; some domains
     may want years and version numbers as tags)
   - Drop single-character tokens
   - Drop tokens under a minimum length (default: 3 chars)

5. **Normalize:** apply §6 rules.

6. **Cap:** soft limit of K tags per entity (default K=8).
   Beyond K, keep the K most-frequent in the content. Hard
   limit: 16 (write fails above this — should never trigger
   in practice).

7. **Write:** as JSON array on the entity's `tags` column.

**Re-generation:** if an entity's primary content is
updated, tags may be regenerated. Initial implementation:
tags are *append-and-dedupe* on content updates (no removal
of previously-applied tags). Future iteration may add
full regeneration with explicit removal.

## §5 Operations

The tag protocol exposes operations via a shared helper
module (e.g., `theory_x/tag_protocol/`). All writers and
readers across the substrate call into this module rather
than parsing JSON columns inline.

**Core operations:**

| Operation | Description |
|---|---|
| `generate(content)` | Run the §4 pipeline; return a list of normalized tags |
| `read(entity_table, entity_id)` | Read tags for an entity; return Python list |
| `write(entity_table, entity_id, tags)` | Replace the tags array (normalizes input) |
| `add(entity_table, entity_id, tags)` | Append tags (deduplicated; normalizes input) |
| `remove(entity_table, entity_id, tag)` | Drop a tag; no-op if absent |
| `query(tag, entity_table=None)` | Find entities containing a tag; if entity_table is None, search all taggable tables |
| `vocabulary(entity_table=None)` | Return distinct tags with frequency counts |
| `merge(old_tag, new_tag)` | Rename a tag everywhere it appears (across all surfaces) |

**Invariants enforced by the module:**

- Every tag written passes §6 normalization
- Tags in storage are always JSON arrays of strings
- Empty arrays are valid (entity may have no tags)
- Tag operations never raise on empty/missing arrays
- Cross-surface operations (query without entity_table; merge) iterate over all taggable tables uniformly

## §6 Normalization

Every tag passes through normalization before being written.
Raw user/auto-generated strings are never stored.

**Rules (applied in order):**

1. Lowercase the entire string
2. Trim leading and trailing whitespace
3. Replace internal whitespace runs with a single hyphen
4. Strip non-alphanumeric except hyphens (`a-z 0-9 -`)
5. Collapse consecutive hyphens into one
6. Strip leading and trailing hyphens
7. Reject if empty
8. Reject if length < 2 characters
9. Truncate to 40 characters if longer (no rejection)

**Examples:**

| Input | Normalized |
|---|---|
| `"Cognition Pattern"` | `"cognition-pattern"` |
| `"  AI Research!  "` | `"ai-research"` |
| `"epistemology / theory-of-mind"` | `"epistemology-theory-of-mind"` |
| `"Phase-29"` | `"phase-29"` |
| `"A"` | (rejected — too short) |
| `""` | (rejected — empty) |
| `"a---b"` | `"a-b"` |

**Enforcement:** the `generate`, `write`, and `add` operations
all run through the same normalize function. There is no path
into the tags column that bypasses normalization.

## §7 Curation

The vocabulary is observable but uncurated by default.
Curation runs out-of-band, on demand.

**Observability primitives:**

- **Frequency view:** `vocabulary()` returns distinct tags with
  occurrence counts across all taggable tables, sorted by
  frequency descending. Low-frequency tags are inspection
  candidates (likely typos, one-offs, or genuinely rare
  concepts that may want consolidation).

- **Near-duplicate detection:** a `near_duplicates(threshold=2)`
  helper returns pairs of tags within a Levenshtein distance
  threshold. Surfaces probable typos: `cognition` vs `cogniton`,
  `memory` vs `memories`.

- **Cluster-similar detection** (advanced; can be deferred):
  tags whose tagged entities have high cosine similarity in
  embedding space are likely conceptual duplicates. E.g., if
  every entity tagged `memory` is also tagged `recollection`,
  they likely overlap semantically and one should be canonical.

**Merge operation:**

```
merge(old_tag, new_tag)
```

Replaces `old_tag` with `new_tag` across every taggable
table. Implementation: iterate over each taggable table,
for each row where `tags` contains `old_tag`, rewrite the
JSON array with `old_tag` swapped to `new_tag` and dedupe.

**Initial deployment:**

Curation is manual. Jon runs the vocabulary view on demand;
inspects merge candidates; performs merges via the helper.

**Future deployment:**

Hooks into Phase 32 RecursiveSelfSpec (§5 row 15 of
DOCTRINE). NEX detects drift autonomously and surfaces
merge proposals through the same review channel that
problems-flagged-for-review use. Jon approves; NEX
performs the merge. This closes the loop on tag-protocol
self-maintenance.

## §8 Cross-surface invariants

The protocol's load-bearing invariants:

1. **One vocabulary across all surfaces.** No DB-local
   dialects. A tag means the same thing wherever it appears.

2. **A tag's spelling is its identity.** `cognition` and
   `Cognition` and `cogniton` are three different tags
   (the first two would be unified by normalization §6;
   the third is a typo that vocabulary curation §7 addresses).

3. **Tag operations are uniform across surfaces.** The same
   write pipeline, the same read pipeline, the same normalize
   function. No table has special tag semantics.

4. **No raw tags reach storage.** Every path into a `tags`
   column flows through the helper module, which enforces
   normalization.

5. **Tags are content-derived by default, not state-derived.**
   A tag should describe what an entity *is about*, not what
   state it is in. State belongs in dedicated columns or table
   membership.

## §9 Test Plan

**Unit tests** (`tests/test_tag_protocol.py`):

1. Normalize: lowercase conversion
2. Normalize: whitespace trim and internal collapse to hyphens
3. Normalize: punctuation strip
4. Normalize: consecutive hyphen collapse
5. Normalize: leading/trailing hyphen strip
6. Normalize: empty rejection
7. Normalize: single-character rejection
8. Normalize: length truncation at 40 chars
9. Generate: produces tags from content via TriggerDetector stopword strip
10. Generate: tag count cap respected (soft K, hard 16)
11. Read: returns Python list for valid JSON
12. Read: returns empty list for default `'[]'`
13. Write: replaces tags array, normalizes input
14. Add: appends, deduplicates
15. Remove: drops tag, no-op if absent
16. Query: finds entities containing tag in one surface
17. Query: finds entities across all surfaces when entity_table=None
18. Vocabulary: returns distinct tags with counts
19. Near-duplicates: returns pairs within Levenshtein threshold
20. Merge: renames tag across all surfaces, dedupes per-row

**Integration tests:**

- Insert a belief; verify auto-tags appear in the column
- Insert a problem; verify auto-tags appear
- Query a tag that appears on both surfaces; verify both come back
- Merge a tag; verify it disappears everywhere and the new tag appears
- Run vocabulary; verify counts match expected frequencies

## §10 Open Items / Future Phases

**Secondary index table** — `tag_index(entity_type, entity_id, tag)`
per DB, populated via writers. Promotion path when inline JSON
scans become a bottleneck. Inline JSON stays source of truth.
Not built initially.

**RecursiveSelfSpec hook** — NEX detects vocabulary drift and
surfaces merge proposals through the review channel. Wired in
Phase 32+.

**Tag-based context in prompt-building** — NEX's prompt
builder gains access to tag context for entities pulled into
the prompt. Tags arrive alongside content; NEX uses them to
contextualize its voice. Wired post-Phase-25b, as the prompt
builder evolves.

**Tag co-occurrence graph** — treat tags as a small concept
network (nodes = tags, edges = co-occurrence frequency).
Enables clustering, conceptual neighborhood retrieval, and
emergent vocabulary structure. Deferred; consider after
vocabulary stabilizes through real-world use.

**Multi-content extraction** — currently tags are generated
from a single primary content field per surface. For surfaces
with multiple content-bearing fields (e.g., problems have
title + description + observations), evaluate whether to
include all fields, weight them, or extract per-field tags.

**Cap policy for tag count** — soft K=8 is initial default;
observe and adjust based on whether entities are getting too
few tags (under-described) or too many (noise).
