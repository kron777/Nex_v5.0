## T10 ("explain how transformers work") Gap-Gate Diagnosis

**Verdict: NOT a misfire.** The gap gate fired correctly. Step-by-step trace:

1. **EC classification:** Technical register — correct.
2. **Membrane routing:** OUTSIDE (no self-inquiry keywords in query).
3. **Retrieval pool:** SQL `LIMIT 200` with `ORDER BY tier ASC, confidence DESC`. All 200 slots are filled by `source='spectrum'` beliefs (200 spectrum beliefs, tier=1, confidence=1.0). Other OUTSIDE sources exist but sit beyond the cutoff.
4. **Side filter:** spectrum not in `_INSIDE_SOURCES` → all 200 pass the OUTSIDE filter.
5. **Keyword scoring:** Query tokens `{'transformers', 'work', 'explain'}`. Only "explain" matches 1 spectrum belief ("…does not explain me…"). "transformers" has zero representation in any belief.
6. **belief_count:** `belief_text.count("- [Tier") = 1`. Threshold: `CHAT_GAP_MIN_BELIEFS = 2`. Technical ∉ `_ALLOW_THIN_REGISTERS`. Gap gate fires.

The belief graph contains zero ML/world-knowledge beliefs. The graph is entirely introspective and philosophical (spectrum, fountain_insight, synergized, koan, tao — all existential in content). The system honestly refused to answer a topic with no representation. This is correct behaviour, not a bug.

**Secondary finding — retrieval pool monoculture:**  
The `LIMIT 200` pool is 100% spectrum on every query. Spectrum fills all 200 candidate slots (200 beliefs × tier=1, conf=1.0), shadowing tao (20 beliefs), koan (39), precipitated_from_dynamic (68), and other non-spectrum OUTSIDE sources. Even on queries those sources would match, they are unreachable because the pool fills before they're considered. This is a separate retrieval design issue, not the cause of T10's deflection. Tracked as fix C2 (LIMIT 200 → 500).

## Three Doctrine-Level Findings

**1. Registered ≠ wired ≠ logging — three separate gates.**  
Today's ports passed registration but some skipped the call site and/or the log write. §6 currently has "called per /api/chat" as an implicit requirement but does not make it a distinct, checkable gate separate from registration. Five of eight nodes failed this gate. §6 needs an explicit "called per-request from chat path" criterion — registration alone is not sufficient evidence that a node is active.

**2. Log path contract was not a ship-condition.**  
§6 #5 ("log file fills") is written as a verification step, but the process did not require verifying it before shipping. Five of eight nodes have no write site. The criterion must be enforced at PR time: a node PR must include the constant, the write site, and a truncate entry in the pre-validation setup. Absent those three, the node is not shippable.

**3. Retrieval monoculture.**  
When one source class dominates the candidate pool by tier+confidence, other sources become unreachable regardless of relevance. This is a new anti-pattern: a pool of 200 that only ever draws from one source is not a pool — it is a single-source retrieval with extra steps. §8 should name this pattern and require a post-seeding pool audit whenever a source exceeds 100 beliefs at tier ≤ 2.
